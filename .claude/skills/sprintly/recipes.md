# sprintly recipes

Concrete flows you can paste with minor tweaks. All assume `sprintly` from PATH (or `~/.sprintly/bin/sprintly`).

Reference IDs used below come from `sprintly projects` / `statuses` / `types` / `sprints` and will differ between environments. Discover them once per session and reuse.

> **Formatting reminder.** Body of `--description` and `--notes` is rendered as HTML by the web UI. Always wrap content in primitive tags (`<p>`, `<h2>`, `<ul><li>…</li></ul>`, `<code>`, `<pre>`, `<b>`, `<i>`, `<br>`). Markdown (`**bold**`, `# heading`, backticks, `- list`) renders as raw syntax — do not use it. Plain unwrapped text is acceptable for one-line notes but `<p>…</p>` is preferred.

## 1. Create a task in the current sprint, assigned to me

`me` resolves to the signed-in user on `issue create` and `issue update` too — no need to look up the numeric ID first.

```bash
sprintly issue create \
  --project 8 --type 1 --sprint 8 --assignee me \
  --priority 4 --subject "Investigate flaky build" \
  --description "<p>Reproduce locally, isolate the failing step, file a follow-up if the fix is non-trivial.</p>"
```

## 2. Move issue N to In Progress

`in progress` status ID comes from `sprintly statuses`. In a fresh sprintly env it is typically `2`, but verify.

```bash
sprintly issue update --id N --status 2 --notes "<p>Picking this up.</p>"
```

## 3. List my open issues across all projects

```bash
sprintly issue list --assignee me --status open --all \
  --sort priority:desc,updated_on:desc \
  --columns id,project,type,status,priority,subject
```

## 4. Move issue to Review with a journal comment

```bash
# Discover the status ID once
sprintly statuses
# Then transition with a short HTML summary
sprintly issue update --id 39 --status 6 --notes \
  "<p>Done, please review.</p><ul><li>Skill files under <code>.claude/skills/sprintly/</code></li><li>Smoke test green</li></ul>"
# Verify the note landed by reading the issue's comments back
sprintly issue show 39 --comments
```

## 5. Batch-create from JSON

`--json-stdin` reads exactly one JSON object. Loop in your shell for batches.

```bash
while IFS= read -r line; do
  echo "$line" | sprintly issue create --json-stdin \
    --project 8 --type 1 --sprint 8 --assignee 9 --priority 4
done <<'EOF'
{"subject":"Add /healthz endpoint","description":"<p>Lightweight liveness probe; no DB hit.</p>"}
{"subject":"Cover login flow with tests","description":"<p>Happy path only for now.</p>"}
EOF
```

The `description` field in each JSON object follows the same HTML rule as `--description` / `--notes`.

CLI flags override JSON keys, so constants (project/type/sprint/assignee/priority) stay outside the JSON.

## 6. Inspect custom fields, then filter / set them

```bash
# Discover IDs and value sets
sprintly custom-fields --project 8 --type 1

# Filter issues by a custom field
sprintly issue list --project 8 --cf 66=2026.07.0 \
  --columns id,subject,cf:66

# Set a custom field on update
sprintly issue update --id 39 --cf 66=2026.07.0
```

## 7. Smoke test (matches #39 acceptance criterion)

End-to-end create → update → list, in one session.

```bash
# Create
sprintly issue create \
  --project 8 --type 1 --sprint 8 --assignee 9 --priority 4 \
  --subject "Skill smoke test" \
  --description "<p>Generated by the sprintly skill smoke test.</p>"

# Capture the new ID from the create output, then transition + comment
sprintly issue update --id "$NEW_ID" --status 2 --notes "<p>Transitioning via skill.</p>"

# Verify it shows up under "me / open"
sprintly issue list --assignee me --status open --subject "Skill smoke test" \
  --columns id,status,subject

# Close it
sprintly issue update --id "$NEW_ID" --status 3 --notes "<p>Smoke test complete.</p>"
```

If every command succeeds and the listing reflects the transition, the skill is sufficient for an LLM to drive without extra hand-holding.

## 8. Find a numeric user id by name

Useful when `me` does not apply (e.g. setting `--watcher` on an issue, or assigning to someone else).

```bash
sprintly users --name larson --columns id,login,name,email
```

Then plug the id into the create/update call:

```bash
sprintly issue update --id 39 --watcher 9
```

## 9. List visible wiki pages of a project

```bash
# By slug
sprintly wiki list --project spr

# Walk a long index with client-side pagination
sprintly wiki list --project spr --limit 50 --offset 50
```

## 10. Read a wiki page, optionally pinned to a version

```bash
# Current revision
sprintly wiki show --project spr --title "Release notes"

# A specific historical revision, with attachments table
sprintly wiki show --project spr --title "Release notes" \
  --version 3 --with-attachments
```

## 11. Upsert a wiki page (create or update by title)

`wiki edit` is a single command for both create and update — the backend keys pages by title. Body source is exactly one of `--text` / `--text-file` / `--text-stdin`.

```bash
# Inline
sprintly wiki edit --project spr --title "Release notes" \
  --text "<p>2026-07-01: shipped #54.</p>" \
  --comments "ship note for sprint 1"

# From a file, set a parent page
sprintly wiki edit --project spr --title "Auth runbook" \
  --text-file ./auth-runbook.html --parent "Runbooks"

# From a pipeline (e.g. generator output)
some-generator | sprintly wiki edit --project spr \
  --title "Daily report" --text-stdin
```

Body follows the same primitive-HTML rule as issue descriptions / notes — Markdown is not interpreted.
